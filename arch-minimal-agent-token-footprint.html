1 · The Concept, In My Own Words

Every headless agent turn pipeline-worker runs today pays two fixed costs before it does any useful work: the tool-use system prompt the underlying CLI injects (Claude Code's full default system prompt, or Copilot CLI's built-in agent prompt), and whatever tool-use documentation the CLI generates for every tool in the invocation's allowlist. For three specific turns — figuring out what a diff is for (captureIntent), fixing merge conflicts, and diagnosing/fixing a failed CI pipeline — you want that fixed cost pushed as close to zero as the CLIs allow, on both the claude and copilot adapters.

Why it matters: these three turns run automatically, on every single run of the tool, often several times per run (the conflict and CI-fix loops can retry up to config.maxFixAttempts times each). Token cost here isn't a one-off — it's a per-invocation multiplier on top of pipeline-worker's baseline usage, and it's the one part of the cost that's pure overhead: none of it goes toward the actual diagnosis or fix. The v2 decision narrows this further for the CI/pipeline-failure stage: the tool budget stays untouched there because the stage cannot do its job without it — only the system-prompt overhead is in play.

2 · What the Code Does Today

Today, tool scoping and system-prompt handling both funnel through the shared AgentAdapter interface (src/agent/types.ts), but the two concrete adapters honor different parts of it:

  • claude.ts forwards allowedTools straight to --allowedTools, but never sets any system-prompt flag — every invocation gets Claude Code's full default system prompt on top of the task prompt.
  • copilot.ts always passes --allow-all-tools and ignores allowedTools/model entirely (per its own code comment); it has no way to touch the system prompt at all.

Only one of the three stages is actually read-only by design: captureIntent.ts sets allowedTools: ['Read','Grep','Glob','Bash(git diff:*)'] because it must never mutate the worktree the later apply/commit steps depend on. The other two are edit loops on purpose: orchestrate.ts's conflict-apply step and watchPipeline.ts's merge-conflict/CI-fix loops run with permissionMode: 'acceptEdits' and no allowlist, and the CI-fix/merge-conflict turns additionally get an MCP config (writeAgentMcpConfig) so the agent can pull failed-job logs itself — the prompt text never contains the failure content.

Gap found during review: that MCP config only actually reaches the agent on the claude adapter (via --mcp-config). copilot.ts drops the same mcpConfigPath argument entirely — it only logs a console.error warning — because Copilot CLI was believed to have no per-invocation MCP flag. Today, under copilot, the forge MCP server is reachable only if a human has separately registered it in the fixed ~/.copilot/mcp-config.json file.
